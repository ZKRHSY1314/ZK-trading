Task 1: Refactor rule scoring semantics

Problem:
backend\app\rules\engine.py currently adds rule weight when any rule passes. That means a constitution or hard risk-control rule can add large positive score merely because it did not block the stock. This is wrong.

Required behavior:
- Separate rule purposes:
  - constitution / hard_block rules: only block or pass, never add positive score.
  - risk rules: may block, warn, or reduce confidence, but should not create a strong buy signal by themselves.
  - strategy rules: can add positive score when passed.
  - data-quality rules: can cap tier or downgrade confidence.
- A stock should become strong only because strategy evidence is strong enough and no hard block is triggered.
- If any hard_block rule fails, tier must be rejected regardless of positive strategy score.
- Unknown rules should not add score.
- Preserve existing CandidateDecision / RuleHit compatibility unless a small model extension is clearly needed.

Acceptance tests:
- A snapshot that only passes constitution_no_high_position must not become strong.
- A snapshot that fails a hard_block rule must be rejected even if strategy rules pass.
- A passed risk rule must not by itself raise a candidate into strong.
- Unknown rules return a safe non-positive result.

Likely files:
- backend\app\rules\engine.py
- backend\configs\rules.yaml
- backend\tests\test_rules_engine.py
- backend\tests\conftest.py if fixtures are useful

Task 2: Add dynamic A-share limit-up threshold logic

Problem:
backend\app\strategies\dengzhan.py uses min_limit_up_pct=9.9 as a fixed proxy. This ignores board differences, ST / risk-warning symbols, and future exchange rule changes.

Required behavior:
- Add a small utility module, for example backend\app\data\price_limits.py or backend\app\data\symbols.py, that determines daily up/down limit thresholds by symbol and metadata.
- Support at least:
  - Shanghai/Shenzhen main board normal stocks: 10% band.
  - ChiNext / STAR Market normal stocks: 20% band.
  - Beijing Stock Exchange normal stocks: 30% band.
  - ST / risk-warning stocks: conservative configurable band.
- Do not hardcode everything inside DengZhanSignals. The thresholds should be testable and configurable enough for later exchange-rule updates.
- Include effective-date awareness or a documented config point because 2026 rule updates may change some risk-warning handling. Current trading-date behavior should not silently assume a future rule is already active.
- If required metadata is missing, return data_warning / observe behavior rather than strong.

Acceptance tests:
- 600xxx / 000xxx normal stock uses 10% threshold.
- 300xxx / 688xxx normal stock uses 20% threshold.
- 8xx / 920xxx Beijing-style stock uses 30% threshold where supported by symbol normalization.
- ST / risk-warning metadata uses the conservative configured threshold.
- Missing pct_change cannot pass limit-up logic.
- Near-limit-up should not equal confirmed limit-up unless explicitly configured.

Task 3: Use adjusted and rolling highs for low-position judgment

Problem:
backend\app\data\akshare_provider.py requests daily bars with adjust="" and snapshot_builder uses all-time raw high. This makes historical-high and low-position judgments unreliable after splits/dividends and across very long listing histories.

Required behavior:
- Prefer forward-adjusted daily bars for strategy judgment, unless there is a clear project reason to keep raw bars separately.
- Keep the data source and adjustment mode visible in snapshot metadata.
- Add rolling high fields to metadata, at minimum:
  - rolling_high_250
  - rolling_high_120 if cheap to compute
  - high_window_used
  - adjust_mode
- Low-position judgment should use the configured rolling window high first, not always all-time high.
- If there is not enough history, return a conservative reason and avoid overconfident strong signals.
- Do not fabricate prices when data is missing.

Acceptance tests:
- SnapshotBuilder records adjust_mode in metadata.
- Low-position logic can use rolling_high_250 from metadata.
- If rolling high is missing or invalid, signal returns a safe failure or data warning.
- A stock below configured rolling high ratio can pass low-position check.
- A stock above configured rolling high ratio fails low-position check.

Likely files:
- backend\app\data\akshare_provider.py
- backend\app\data\snapshot_builder.py
- backend\app\strategies\dengzhan.py
- backend\configs\rules.yaml
- backend\tests\test_snapshot_builder.py
- backend\tests\test_dengzhan_signals.py

Task 4: Enforce market-cap filters and data-quality downgrades

Problem:
rules.yaml contains min_market_cap_billion / max_market_cap_billion, but the strategy path does not fully enforce market_cap_billion. Also fallback data can still flow into planning with too much confidence.

Required behavior:
- MarketSnapshot already has market_cap_billion. Ensure snapshot construction populates it when available from profile/provider data.
- DengZhan low-position limit-up strategy must enforce configured market-cap bounds when data exists.
- If market cap is missing, add a data warning and do not allow strong tier from that rule alone.
- If data_quality is fallback_profile or realtime_quote_fallback, cap tier at watch or reject unless explicitly configured otherwise.
- The simulation planner should keep fallback-profile and low-quality data as observe-only or reduced confidence.

Acceptance tests:
- Market cap below min fails or downgrades.
- Market cap above max fails or downgrades.
- Missing market cap adds warning and cannot produce strong by itself.
- fallback_profile cannot produce strong buy plan.
- realtime_quote_fallback cannot produce strong buy plan without daily-bar confirmation.
