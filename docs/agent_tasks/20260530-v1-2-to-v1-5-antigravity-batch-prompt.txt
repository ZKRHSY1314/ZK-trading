# V1.2 Task Packet: Event-Driven Historical Backtesting

## Goal

Replace the current label/statistics-style backtest skeleton with a real daily-bar event-driven historical replay engine. The backtest should simulate signals, entries, exits, costs, T+1 constraints, position sizing, and metrics over historical cached bars.

## Scope

In scope:

- Add an event-driven backtest module under backend/app/backtest/.
- Use cached daily bars where available.
- Support a date range, initial cash, strategy config, and bounded symbol universe.
- Reuse current rule engine, market snapshots, simulation constraints, and v1.1 data-quality gates.
- Generate auditable trades, equity curve, and metrics.
- Add API endpoint(s), CLI automation mode, tests, docs, and a compact frontend panel if reasonable.

Out of scope:

- Intraday tick backtesting.
- Real broker execution.
- Paid/private data integration.
- Automatic parameter updates.
- Claims of profitability.

## Requirements

1. Backtest engine
   - Create a module such as backend/app/backtest/engine.py.
   - Inputs should include:
     - start_date
     - end_date
     - symbols or candidate source
     - initial_cash
     - max_positions
     - per-symbol position cap
     - strategy/rule config reference
   - Replay trading days in chronological order.
   - At each day:
     - build snapshots from historical bars, not latest quote fallback,
     - evaluate rules,
     - create simulated entry/exit decisions,
     - enforce T+1,
     - apply lot size, fees, stamp tax, slippage,
     - update cash, positions, realized P/L, and equity.
   - Do not trade a symbol if historical data is missing or marked low quality.

2. Data source
   - Prefer daily_bar_cache.
   - If cache is insufficient, return an explicit insufficient_data result.
   - Do not silently fetch live latest quotes for historical bars.
   - Do not fabricate missing bars.

3. Strategy behavior
   - Use current RuleEngine and DengZhanSignals unless a thin backtest adapter is needed.
   - Strong candidates may produce simulated entries only if all hard risk gates pass.
   - Watch candidates should not automatically enter unless explicitly configured.
   - If v1.1 fallback/data-quality caps are triggered, treat as observe/no trade.

4. Exit logic
   - Implement a conservative first version:
     - stop loss if close or low breaches stop loss,
     - take profit if high or close reaches target,
     - max holding days,
     - hard risk exit if rules become blocked.
   - Clearly document simplified assumptions.

5. Metrics
   - total_return
   - annualized_return if enough days
   - max_drawdown
   - win_rate
   - profit_loss_ratio
   - trade_count
   - average_holding_days
   - exposure_ratio
   - skipped_due_to_data_count
   - rejected_by_risk_count

6. Persistence
   - Store backtest runs, trades, daily equity, and summary metrics in SQLite.
   - Each run must record config_json, data_source, start/end dates, created_at, and status.

7. API / CLI
   - Add endpoints similar to:
     - POST /api/backtest/runs
     - GET /api/backtest/runs?limit=20
     - GET /api/backtest/runs/{id}
   - Add CLI automation mode such as:
     - python scripts/automation_loop.py --mode backtest --max-cycles 1
   - Keep the mode read/simulation-only.

8. Frontend
   - Add or extend a compact Backtest panel.
   - Show latest run, metrics, trade count, skipped data count, and disclaimer.
   - Avoid raw JSON by default.

9. Tests
   - Use deterministic fixture bars.
   - Test T+1 sell restriction.
   - Test fee/stamp/slippage application.
   - Test insufficient history returns no fabricated trades.
   - Test hard-blocked symbols do not enter.
   - Test metrics calculation.

# V1.3 Task Packet: Portfolio Risk and Market Regime

## Goal

Add portfolio-level risk controls and market environment filtering so the system can reduce or stop new simulated entries in weak or extreme market conditions.

## Scope

In scope:

- Add a market_regime module.
- Add portfolio/account risk gates.
- Integrate gates into simulation planning and backtest entry logic.
- Add API/CLI/frontend visibility and tests.

Out of scope:

- Real account synchronization.
- Real broker positions.
- Automatic live order throttling.
- Claims that regime classification predicts the market.

## Requirements

1. Market regime service
   - Create backend/app/market_regime/.
   - Classify market environment using public index/cached data where available.
   - Start with simple deterministic features:
     - major index trend vs moving average,
     - index 5/20 day return,
     - breadth if available,
     - recent limit-up / limit-down counts if already in local data,
     - volatility proxy.
   - Output:
     - regime: strong, neutral, weak, extreme_risk, insufficient_data
     - confidence
     - reasons
     - data_quality
     - updated_at

2. Portfolio risk gates
   - Add config values for:
     - max_total_exposure
     - max_single_position
     - max_new_positions_per_day
     - max_daily_loss
     - max_drawdown_stop
     - consecutive_loss_cooldown
   - Apply gates to simulation planner and backtest entry decisions.
   - Weak regime should reduce position ratio.
   - Extreme risk regime should stop new entries and produce observe-only plans.
   - Consecutive losses should trigger cooldown/observe mode.

3. Persistence
   - Store market regime snapshots.
   - Store portfolio risk state or derive it from simulation/backtest records.

4. API / CLI
   - Add endpoints similar to:
     - GET /api/market-regime/latest
     - POST /api/market-regime/refresh
     - GET /api/risk/portfolio-state
   - Add CLI mode such as:
     - --mode market-regime
   - Do not mutate strategy weights automatically.

5. Frontend
   - Add a compact market regime / portfolio risk panel.
   - Show regime, risk gate status, reasons, and suggested posture.
   - Use clear labels such as observe, reduce, stop_new_entries.

6. Tests
   - Weak regime reduces position ratio.
   - Extreme risk blocks new simulated entries.
   - Missing regime data does not create aggressive plans.
   - Portfolio exposure cap prevents new entry.
   - Consecutive loss cooldown creates observe-only plans.

## Acceptance Checks

Expected:

- /health remains live_trading_enabled=false.
- Simulation/backtest entries respect market regime and portfolio gates.
- Frontend shows regime/risk state without implying guaranteed prediction.
- Handoff file created: docs/agent_tasks/stage_v1_3_handoff.md.

## Scope

In scope:

- Enhance candidate monitoring using existing public/latest quote paths.
- Add alert lifecycle and deduplication.
- Add operator action logging.
- Add API/CLI/frontend updates.

Out of scope:

- Tick-level HFT.
- Broker screen control.
- Real order placement.
- Real-time guaranteed data feeds.

## Requirements

1. Monitoring lifecycle
   - Add or extend states:
     - discovered
     - watching
     - alerted
     - simulated_plan_ready
     - ignored
     - rejected
     - review_required
   - Persist lifecycle transitions with timestamps and reasons.

2. Alert logic
   - Deduplicate repeated alerts for the same symbol/reason within a configurable cooldown.
   - Escalate when a watch candidate becomes strong with clean data and risk gates pass.

     - rule hits,
     - hard-block status,
     - data-quality status,

     - stop loss / target if available,

3. Operator actions
   - Add actions:

     - ignore_today
     - add_to_review

4. API / CLI
   - Add endpoints similar to:
     - GET /api/monitoring/alerts/latest
     - POST /api/monitoring/alerts/{id}/action
     - GET /api/monitoring/lifecycle?symbol=...
   - Add or refine CLI mode:
     - --mode monitor
   - Ensure automation can run one monitoring cycle without needing real broker access.

5. Frontend
   - Add an intraday alert/lifecycle panel.
   - Show only user-relevant fields by default.
   - Hide raw debug JSON unless developer mode is enabled.
   - Use Chinese labels for key states and actions where the existing UI pattern allows.

6. Tests
   - Alert dedupe works.
   - Stale data de-escalates.
   - Hard risk creates reject/review rather than buy plan.
   - Operator actions are logged.

# V1.5 Task Packet: AI-Assisted Review and Parameter Proposal

## Goal

Add an AI-assisted review layer that can summarize signals, generate post-market reviews, and propose parameter calibration changes, while keeping all AI output behind validation, sample-size, out-of-sample, and human-approval gates.

## Scope

In scope:

- Add a provider-neutral AI interface if not already sufficient.
- Add deterministic disabled/mock provider behavior for tests.
- Add AI review records and audit summaries.
- Add parameter proposal objects and approval workflow.
- Validate proposals using v1.2 backtest and v1.3 risk metrics.
- Add frontend AI review/proposal panel.

Out of scope:

- AI direct order placement.
- AI direct production config mutation.
- AI enabling live trading.
- Storing API keys in code.
- Exposing API keys to frontend.
- Full autonomous trading.

## Requirements

1. AI provider interface
   - Extend or implement a provider-neutral interface such as:
     - analyze_signal(context)
     - summarize_daily_review(context)
     - propose_parameter_change(context)
     - review_risk(context)
   - Providers may include disabled/mock/local placeholders.
   - Real external provider support must read API keys only from environment variables.
   - If no API key exists, return disabled/unavailable cleanly.

2. AI audit log
   - Store:
     - provider name
     - model name if available
     - input summary, not huge raw prompts
     - output summary
     - final recommendation
     - risk flags
     - created_at
     - linked symbol/run/proposal ids
   - Do not store credentials.

3. Parameter proposals
   - Add proposal type:
     - proposed config/rule changes
     - rationale
     - expected effect
     - risk notes
     - status: draft, validation_failed, validation_passed, pending_human_review, approved_for_simulation, rejected
   - AI may create proposals but may not apply them directly.
   - Any proposal must run v1.2 backtest validation.
   - Require:
     - enough trade count,
     - non-worse max drawdown,
     - non-worse win rate,
     - non-worse profit/loss ratio,
     - out-of-sample check if enough data exists,
     - no hard risk override.

4. Dual-review optional structure
   - If easy, prepare fields for primary_review and risk_review.
   - It is acceptable to keep the second provider disabled/mock for now.
   - Any disagreement should downgrade to human review.

5. API / CLI
   - Add endpoints similar to:
     - POST /api/ai/reviews/daily
     - GET /api/ai/reviews/latest
     - POST /api/ai/proposals
     - GET /api/ai/proposals?limit=20
     - POST /api/ai/proposals/{id}/validate
     - POST /api/ai/proposals/{id}/approve-for-simulation
     - POST /api/ai/proposals/{id}/reject
   - Add CLI mode such as:
     - --mode ai-review
   - Approval means simulation-only, not live trading.

6. Frontend
   - Add an AI review/proposal panel.
   - Show:
     - signal summary,
     - main evidence,
     - main risks,
     - validation status,
     - human review status.
   - Display a disclaimer:
     - AI output is for review and simulation only, not investment advice.
   - Do not show raw prompts by default.

7. Tests
   - AI cannot change enable_live_trading.
   - AI cannot call or create live order endpoints.
   - Proposal validation rejects insufficient sample size.
   - Proposal validation rejects worse drawdown.
   - Approved proposal is simulation-only.
   - Missing API key is handled gracefully.

## Acceptance Checks

Expected:

- AI review works in disabled/mock mode without external credentials.
- No API key is committed or exposed.
- AI proposal cannot directly mutate production rule config.
- Proposal validation uses backtest/risk gates.
- /health remains live_trading_enabled=false.
- Handoff file created: docs/agent_tasks/stage_v1_5_handoff.md.
