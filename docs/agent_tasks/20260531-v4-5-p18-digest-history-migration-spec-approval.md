# V4.5-P18 Digest History Migration Spec Approval Metadata

## Goal

Add operator approval metadata for a dry-run verified `screen_readiness_digest_history` migration spec.

This stage records review evidence only. It does not execute SQL, create tables, run migrations, write migration files, write digest history records, create downloads, execute commands, inspect windows, capture pixels, run OCR, click/type, connect brokers, or affect live trading.

## Implemented Scope

- Backend exposes `POST /api/screen-monitoring/readiness-health/history-migration-spec/approve`.
- Backend exposes `GET /api/screen-monitoring/readiness-health/history-migration-spec/approvals`.
- `ScreenMonitoringService.approve_screen_readiness_digest_history_migration_spec()` first runs the V4.5-P17 dry-run verifier.
- Passing specs record approval metadata as an event in the existing `events` table.
- Failed specs return `approval_blocked` and do not create approval metadata.
- Screen readiness timeline includes approval metadata events.
- Frontend V4.5 panel adds "确认草案已审阅" and compact approval/history cards.
- Release checklist and cockpit skill were updated for V4.5-P18.

## Default Behavior

The UI approval action uses the safe default dry-run migration spec generated by the verifier. Approval creates only an audit event. Future migration still requires a reviewed SQLite migration, rollback plan, migration tests, API smoke tests, forbidden tracked-file scan, and explicit release approval.
